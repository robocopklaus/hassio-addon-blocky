# Passive config migration via schema retention

Home Assistant does not auto-migrate add-on options. On a schema change the Supervisor keeps a user's persisted value for every key still in the schema and **silently drops** the rest — there is no upgrade hook, and the add-on cannot safely rewrite `/data/options.json` because HA owns that file and regenerates it from the UI. So the add-on's only migration lever is **passive**: keep a deprecated option key in the schema (so HA never strips it) and have `blocky.gtpl` translate its old shape into the current Rendered config on every render. The `upstreams.start_verify` → `init_strategy: failOnError` mapping is the reference example.

The risk this creates is **silent drop**: a future maintainer sees a deprecated key sitting in the schema, assumes it is dead, deletes it — and every existing user's setting vanishes with no error. The contract checker cannot catch this because it enforces template/schema/translations parity only at the *top level* (it verifies `upstreams` exists, not `start_verify` beneath it), and full nested parity is infeasible — the template aliases nearly every section into a `$var` (`$blocking.denylists`, `$queryLog.fields`, …), so static path resolution would be brittle interpreter-lite work.

**Therefore:** deprecations are recorded in an explicit registry, `scripts/deprecations.json` (`{path, since, replacedBy, fixture}`), and `check-config-contract.mjs` asserts for each entry that the `path` is still present in **both** the `options` and `schema` blocks of `config.yaml`, and that its overlay `fixture` exists. The registry lives outside the schema on purpose — a marker on the schema line would disappear with the line it is meant to protect. Division of labour: the **static check** owns silent-drop (the key stays in schema + options); the **overlay fixture** (e.g. `start-verify-legacy`) owns translation correctness (the old value lands in the new field, proven by `tempio` render + golden diff).

**Considered & rejected:**

1. *Active migration — rewrite `/data/options.json` on startup.* Rejected: HA owns that file and clobbers any rewrite on the next UI change; it fights the platform instead of working with it.
2. *Schema-aware harness — teach `run.mjs` to prune options to the schema before rendering, so existing fixtures catch silent-drop via a golden diff.* Rejected: the signal is an ambiguous golden diff (`failOnError` quietly vanishes) exactly when clarity about the stakes matters most, it needs a structural schema-walk that risks diverging from HA's real filtering, and it changes the semantics of every fixture. The static check instead emits a precise diagnostic naming the dropped key and the users it would affect.
3. *Full nested parity in the contract checker* — rejected: defeated by the template's pervasive `$var` aliasing (see above).
4. *A runtime guard warning when a deprecated key is still set* — deferred, not rejected. It earns its keep only when a removal is actually scheduled (you need users off the key first); the registry's `since` field will drive it at that point.

**Consequence:** Deprecating an option is now a deliberate, three-part act — keep the key in `config.yaml` (options + schema), add the translation to `blocky.gtpl`, register it in `deprecations.json` with an overlay fixture. Removing a deprecated key is a breaking change that requires deleting its registry entry, schema/options keys, template mapping, and fixture together; the static check keeps them in lockstep until then. The Blocky *binary format* surface is out of scope here: in Standard Mode the template absorbs Blocky's own field changes and the render harness's per-fixture `blocky validate` catches breakage on a `BLOCKY_VERSION` bump; in Custom Config Mode the operator owns the YAML (ADR-0004) and relies on Blocky's own warnings plus the changelog. The check tests key *presence* by dotted path (so a deprecated key with a bare-empty default, or a whole object/list subtree, is handled) and that the named fixture's `override.json` actually exercises the key; it deliberately does not validate option *types*, which `blocky validate` and the render goldens already cover.
