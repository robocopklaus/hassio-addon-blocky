# A single local check aggregate; CI stays parallel jobs

The translation pipeline has three verification entrypoints — the **contract checker** (`scripts/check-config-contract.mjs`, static three-file parity, ADR-0003), the **guard test** (`scripts/test-guards.sh`, guard functions vs render goldens, ADR-0004), and the **render harness** (`scripts/render-test/run.mjs`, real `tempio` + `blocky` render vs goldens, ADR-0003). They had drifted into three invocation styles (`node …` / `bash …` / `pnpm run check:render`), the guard test had **no `package.json` alias at all** (only ever run as `bash scripts/test-guards.sh` from CI), and `CLAUDE.md` still claimed *"No automated tests. Testing is manual."* A contributor had no single command to run before committing, and the guard check was easy to forget locally.

**Decision.** Add `pnpm check` (`scripts/check.mjs`) as a single **local** aggregate, and give the guard test a `check:guards` alias so it is no longer orphaned. The aggregate:

- runs **contracts + guards always** — both instant, both any-OS (contracts is dependency-free Node, guards is bash reading committed goldens), both results collected with no short-circuit between them;
- runs **render only if both passed** — bare on Linux; off-Linux it dispatches render into the throwaway Docker container already documented in ADR-0003, but **isolates** that container's `pnpm install` from the host by masking `node_modules` (and the pnpm store) with private volumes, so the Linux-resolved tree never overwrites the developer's platform-resolved one. `run.mjs`'s **Linux-only refusal logic is unchanged** — the aggregate owns the platform decision, the harness still refuses off-Linux. The one seam shared with `run.mjs` is the Docker command itself: it is single-sourced in `scripts/render-test/docker-render.mjs` and used both by the aggregate's dispatch and by `run.mjs`'s printed macOS hint, so the isolated invocation can never drift between the executed path and the copy-paste path.
- treats a render that **cannot run** (off-Linux and Docker unavailable, or a Docker image/daemon error or timeout — distinguished by docker's own 125/126/127 exit codes) as `⚠`, distinct from a `✗` render failure, and **exits non-zero** for it. A green `pnpm check` therefore means all three actually ran and passed — it can never hide an unverified render behind a soft skip.

**CI (`lint.yml`) keeps its three lean parallel jobs and does not run the aggregate.** CI is Linux, so render runs bare there — the Docker dispatch that motivates the aggregate is a macOS-only concern. The jobs are already independent and optimally set up (the contracts job is deliberately dependency-free per ADR-0003; the guard job is bare bash; the render job caches the provisioned binaries), parallel gives faster wall-clock and granular per-check status, and the aggregate's local render-gate (skip render when a fast check is red) is exactly wrong for CI, where every result is wanted on every run. Dev and CI cross the **same three check scripts**, composed differently — one sequential command vs. parallel jobs.

**Scope.** `pnpm check` is the three translation-pipeline harnesses only. `hadolint` and `shellcheck` stay their own CI jobs: they need non-npm binaries that may be absent locally, and pulling them in would reintroduce the "needs extra tooling" friction the aggregate removes.

**Considered & rejected:**

1. *Collapse CI to one `pnpm check` job* — serial instead of parallel, forces pnpm install + binary provisioning onto the whole job (losing ADR-0003's dependency-free contracts job), flattens per-check status, and the local render-gate would hide the render result on any run where a fast check is red.
2. *Route every CI job through `pnpm check:*` aliases* for surface uniformity — adds pnpm + Node setup to the dependency-free and bare-bash jobs purely to call an alias.
3. *Make `run.mjs` soft-skip or auto-Docker itself* — mixes platform dispatch into the harness; ADR-0003 deliberately keeps `run.mjs` Linux-only and refusing (its goldens are authoritative only from the shipped Linux binaries). The dispatch belongs in the aggregate, not the harness.
4. *Exclude render from the aggregate* (contracts + guards only) — the one command would then never cover the most bug-prone check locally.
5. *Let render soft-skip to exit 0 when it cannot run* — false green, the precise failure the gate exists to prevent.

**pnpm's default dependency verification is left on.** An earlier revision set `verifyDepsBeforeRun: false` in `pnpm-workspace.yaml` to stop a no-TTY abort during `pnpm check`. That was a misdiagnosis: the abort came from the Docker render clobbering the host `node_modules` (before the volume isolation above), which left the tree incompatible with the store so pnpm wanted to purge it. The flag is also workspace-**global** — it would have silenced the pre-run check for `release` and `render:update` too, so a stale-lockfile run there (e.g. after a Renovate bump) would use the wrong dependency versions with no warning. With the render isolated, the host tree stays healthy and pnpm's default (`install` — auto-installs before a script) is kept: it preserves that safety net and installs deps up front on a fresh clone, so the bare-Linux render never runs without its `yaml` dependency. CI is unaffected either way — it installs explicitly with `pnpm install --frozen-lockfile` and runs the three scripts directly, not through the aggregate.

**Consequence.** `pnpm check` is the local pre-commit gate; a fully-green run on macOS needs Docker running, and CI remains the authoritative render gate regardless. Adding a fourth check is a two-part act: an entry in `check.mjs` (plus a `check:<name>` alias), and — because CI stays parallel — a matching job in `lint.yml`. The aggregate and CI must stay behaviorally identical per underlying script; they intentionally differ only in composition (sequential vs. parallel) and the local-only render-gate. A future review that proposes "just have CI run `pnpm check`" should stop here: the split is deliberate.
