# Format-coupled guards run only in Standard Mode

ADR-0002 establishes that guards must read the **rendered** config, never re-derive from options. Some guards go further and **text-parse** that rendered YAML — e.g. the query-log guard greps `target:` with `sed`, and the upstreams core guard scans the `upstreams:` block with `awk` for a `default` group with a resolver. Such guards are coupled to the *exact output format the template produces*: fixed indentation, double-quoted keys, list items at a known depth.

That coupling is safe **only** against template-generated configs. In Custom Config Mode the operator supplies a complete, hand-written Blocky YAML, which may legitimately use flow style (`groups: {default: [...]}`), different indentation, anchors, or comments. A format-coupled parser run against it can **false-positive and abort a perfectly valid config** — turning a working setup into a failed boot. A core fail-fast guard that rejects correct input is worse than no guard at all.

**Therefore: guards that text-parse the rendered config gate on Standard Mode** (`! bashio::config.true 'custom_config'`). In Custom Config Mode the operator owns the file; Blocky's own startup logging surfaces upstream and other errors for them.

This is what motivated the first instance. `upstreams` is a core feature (ADR-0002): Blocky needs a `default` group with at least one resolver, or it starts but cannot resolve DNS. We discovered — via the render-test harness fixture `upstreams-empty` — that `blocky validate` **accepts** an empty `upstreams.groups`, so ADR-0002's promised fail-fast never actually happened. The new guard closes that gap, but only in Standard Mode per the rule above.

**Considered & rejected:** (1) *Run format-coupled guards in both modes* (consistent "guards read the rendered config regardless of mode") — rejected because the parser would misfire on YAML shapes it was never built for, and a core guard must never abort a correct config. (2) *Make the guards YAML-aware* (parse with a real YAML tool so any format is handled) — rejected: the container has no `yq`, only `sed`/`awk`/`jq` (JSON), and pulling in a YAML parser for in-container guards is disproportionate; the narrow format-coupling is acceptable precisely because the template generates the format deterministically. (3) *Rely on `blocky validate` alone* — rejected: it does not catch the empty-upstreams case, which is the whole reason this guard exists.

**Consequence:** Every new guard that inspects the rendered config by text-parsing must gate on Standard Mode. The detection lives in `blocky/rootfs/usr/lib/blocky/guards.sh` as pure functions (config path in, exit status out) so it is single-sourced between the runtime guard and `scripts/test-guards.sh`, which asserts it against the committed render-test goldens. Custom Config Mode receives no such protection by design.
