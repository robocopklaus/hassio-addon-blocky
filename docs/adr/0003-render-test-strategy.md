# How the translation pipeline is tested

The core complexity of this add-on is the **translation pipeline**: `blocky.gtpl` turns Home Assistant options into the Rendered config. It is the most bug-prone part (several `fix:` commits harden it) yet had no interface a test could cross short of booting the add-on inside Home Assistant. `check-config-contract.mjs` proves the three-file lockstep (options ↔ schema ↔ translations, and that the template references every top-level key) but **never renders** — nothing exercised the actual translation.

We add a **render harness** (`scripts/render-test/run.mjs`) that crosses the same seam the add-on does at runtime: `options.json → tempio + blocky.gtpl → Rendered config → blocky validate`. It is **additive** to the contract checker, not a replacement — the contract checker still guards parity across files (the `schema` and `translations` blocks never appear in rendered output, so the harness cannot see them).

**Decisions:**

- **Runs bare on the CI runner against the real, pinned binaries.** Rendering is pure text transformation with no runtime filesystem dependency, so the harness downloads the same `tempio` and `blocky` releases the Dockerfile pins and runs them directly on `ubuntu-latest` — exactly how `check-config-contract.mjs` already runs. No image build on the PR path.
- **The Dockerfile is the single version pin.** The harness greps `BLOCKY_VERSION`, `TEMPIO_VERSION`, and `TEMPIO_SHA256_AMD64` out of `blocky/Dockerfile`, so a Renovate bump re-tests the new version's rendering on the same PR. There is no second place to update.
- **Always the shipped Linux binaries, regardless of developer OS.** `tempio` publishes no macOS build, so goldens are authoritative only when produced by the Linux binaries we actually ship. The harness refuses to run off-Linux and points at a throwaway-container one-liner; CI and Linux dev boxes run bare.
- **Full-file golden snapshots are the contract.** Each fixture renders to a committed `expected.yml`; the harness diffs byte-for-byte. A diff catches *unintended* output (the class of the `customTTL` / "invalid combinations" bugs), and byte-level formatting genuinely matters — `config.sh` greps the rendered `target:` with a precise `sed` pattern. The golden diff doubles as the PR review surface for upstream Blocky changes.
- **Fixtures are defaults-overlay merges.** A fixture is a tiny `override.json` deep-merged onto the `options:` block read live from `config.yaml` (arrays replace, objects merge — mirroring how HA composes options). Changing a shipped default re-renders every golden, so drift surfaces as a diff. Defaults stay single-sourced in `config.yaml`.
- **`blocky validate` runs per fixture; default expect-pass.** A passing validate on a degraded config is the *teeth* of ADR-0002's degrade promise ("valid YAML with the broken part removed"). A fixture may drop an `expect-invalid` marker file to assert validate *fails*, pinning the core fail-fast paths (e.g. empty `upstreams`).
- **`--update` regenerates goldens; CI is read-only.** Regeneration is a deliberate local action. CI only ever renders-and-asserts, so it can never silently self-heal a diff.

**Considered & rejected:** (1) *Building the Docker image and running the harness inside it on every PR* — maximally faithful but pays a full multi-arch build per PR and is awkward to run locally; rendering needs none of the container's filesystem. (2) *Render-testing only in the release pipeline against the built image* — cheap, but then it no longer guards PRs, which is the whole point. (3) *Fragment/key assertions instead of full-file goldens* — a fragment check only sees what it is told to look at and misses unintended emissions. (4) *Folding the contract check into the harness* — mixes static three-file parity with real rendering and puts the regex parser next to authoritative output. (5) *Reimplementing tempio's rendering in Node* — would produce goldens that need not match the real binary; the seam exists precisely to test the real translation.

**Consequence:** A `fix:` to the translation lands with the fixture that would have caught it, so coverage compounds toward exhaustive over time instead of being front-loaded. The harness needs a YAML parser (the overlay merge reads `config.yaml` structurally), so the render-test CI job installs dev dependencies — unlike the dependency-free contract job. The first golden set must be generated in a Linux environment before the CI job can pass.
