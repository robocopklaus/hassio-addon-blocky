# Degrade-warn guards read intent from options, outcome from the rendered config

ADR-0002 establishes that guards read what they check from the rendered config, never re-derive it from `options.json`. A **degrade-warn** guard is the one principled exception: it exists to announce the gap between what the operator *asked for* and what the template *actually emitted* (e.g. "you enabled HTTPS but no `certFile` was rendered, so TLS was dropped"). The intent — *did the operator ask for this feature?* — only lives in `options.json`, because the template has just dropped the feature from the rendered config; you cannot reconstruct "requested but dropped" from a config where the feature is simply absent. So a degrade-warn guard reads **intent from `options.json`** and observes **outcome from the rendered config** — it must not re-compute the template's *drop condition* (the real drift risk ADR-0002 names).

This sharpens ADR-0002 for the degrade case without contradicting it. The fail-fast and the path-bearing guards (upstreams core guard, query-log target) read a value that is *present* in the rendered config when the feature is on, so they read it straight from the rendered YAML. The HTTPS/DoH and `single_name_order` guards are different: they fire precisely when the feature is *absent*, so the "present?" signal must be the observed absence in the rendered config, and the "wanted?" signal must come from options.

**Concretely:** instead of `if https.enable && ! has_value cert_file` (which re-implements the template's "drop TLS when the cert is missing" decision in a second language), the guard becomes `if https.enable` (intent, from options) **and** `certFile` absent from the rendered config (outcome, observed — via a pure function in `usr/lib/blocky/guards.sh`, tested through `scripts/test-guards.sh` against the render-test goldens, exactly like the query-log guards in ADR-0004). Same for `single_name_order`: intent from options, "did `singleNameOrder` survive into the rendered config?" observed, not re-derived.

**Considered & rejected:** (1) *Read everything — including intent — from the rendered config* (a naive reading of ADR-0002) — impossible for a drop-warn: "feature never requested" and "feature requested and dropped" produce an identical rendered config, so the warning could never distinguish them. (2) *Leave the two guards re-deriving the drop condition from options* (status quo) — rejected: it duplicates the template's drop logic in a second language, so a change to the template's gating silently desyncs the warning — the exact drift ADR-0002 forbids.

**Consequence:** The "did the feature survive?" half of every degrade-warn guard is a pure, format-coupled function in `guards.sh` (Standard Mode only, per ADR-0004) with a render-test golden pinning it; only the "did the operator ask?" half reads `options.json`. A future contributor who sees the HTTPS guard read `options.json` and reaches for ADR-0002 to "fix" it should stop here: reading intent from options is correct; re-deriving the drop is not.

Because the outcome read is format-coupled, these warnings now gate on Standard Mode (ADR-0004) alongside the upstreams and query-log guards — so they are silent in Custom Config Mode, including the first-run-generate path where the config is still template-derived. (Before this, they read options directly and so fired in both modes — including spuriously in steady-state Custom Mode, where UI options are ignored.) That first-run silence is the accepted cost of the ADR-0004 gating and is consistent with the other format-coupled guards; the alternative — a "was this config generated this boot?" gate that would also re-enable the fatal upstreams guard on custom first-run — is a broader change left to its own decision.

**Revisited 2026-07-04:** An architecture review floated giving the degrade-warn pattern its own module — a `degrade_warn` helper folding the `intent ∧ ¬outcome → warn` tail, pitched to live in `guards.sh` and to "unify" the two intent reads. Declined for now. The pitch's two headline wins are unbuildable. (1) *Lives in `guards.sh`* — every function there takes a config path as `$1` and reads nothing else, which is exactly what lets `test-guards.sh` assert it against goldens that contain no `options.json`; a helper that reads intent from options takes a second input the harness cannot supply and breaks that invariant. (2) *Unifies the intent read* — the two intents are irreducibly different: HTTPS intent is a compound `https.enable || http3.enable` bashio boolean, while `single_name_order` intent must count the array with `jq`, because bashio's `has_value` treats an empty list as "present" (the wrong answer here). They cannot collapse into one read. The only buildable shape is a pairing helper in `config.sh` (never `guards.sh`) taking a named *intent function* + the existing pure outcome function + message lines. For the two current sites that shape raises total line count — two intent fns + two outcome fns + one helper against two self-contained `if` blocks that today carry their ADR rationale inline — and it only *nudges* toward the split rather than enforcing it: a careless intent function can still re-derive the template's drop condition (`https.enable && ! has_value cert_file`), the exact drift this ADR forbids. At two sites, the inline convention reads better than the indirection. **Reopen when a third degrade-warn lands** (rule of three): the pairing helper then pays for itself, and its design — intent-fn + outcome-fn + messages, in `config.sh` — is settled here so it need not be re-derived.
